## 18.1 pip：安装第三方库

Python 强大的地方在于有海量的**第三方库**（别人写好的、可以直接用的代码）。`pip` 是 Python 的包管理工具，从 [PyPI](https://pypi.org)（Python 包索引）下载安装它们。

```bash
pip install requests          # 安装名为 requests 的库
pip install numpy pandas      # 一次装多个
pip install "django>=4.0"     # 指定版本
pip uninstall requests        # 卸载
pip list                      # 查看已安装的库
pip show numpy                # 查看某个库的详细信息
```

> 如果 `pip` 命令不行，试试 `pip3`，或者 `python -m pip install ...`（最稳妥，明确用当前 Python 的 pip）。

安装后就能直接 `import` 使用：

```python
import requests               # 需要先 pip install requests
resp = requests.get("https://api.github.com")
print(resp.status_code)       # 200
```

## 18.2 为什么需要虚拟环境

设想一个问题：项目 A 需要 `numpy 1.20`，项目 B 需要 `numpy 2.0`。如果所有库都装在同一个全局环境里，就会**版本打架**。

**虚拟环境**为每个项目创建一个**独立、隔离**的 Python 环境，各项目的依赖互不干扰。这是专业开发的标准做法，强烈建议每个项目都用虚拟环境。

好处：

- 隔离依赖，避免版本冲突。
- 环境可复现——把依赖清单给别人，对方能装出一模一样的环境。
- 不污染系统的全局 Python。

---

## 18.3 用 venv 创建虚拟环境

`venv` 是 Python 自带的虚拟环境工具，无需额外安装。

### 创建

在项目文件夹里运行：

```bash
python -m venv .venv          # 创建一个名为 .venv 的虚拟环境
```

这会生成一个 `.venv` 文件夹，里面是独立的 Python 解释器和库目录。

### 激活

激活后，`python`、`pip` 命令就指向这个隔离环境了：

```bash
# Windows (PowerShell)
.venv\Scripts\Activate.ps1

# Windows (cmd)
.venv\Scripts\activate.bat

# macOS / Linux
source .venv/bin/activate
```

激活成功后，命令行提示符前面通常会出现 `(.venv)` 字样。此时 `pip install` 装的库只会进入这个环境。

### 退出

```bash
deactivate
```

> **提示**：`.venv` 文件夹不要提交到 git（体积大且和机器相关），应写进 `.gitignore`（本教程根目录的 `.gitignore` 已包含它）。

## 18.4 requirements.txt：依赖清单

如何把“我这个项目需要哪些库”记录下来、方便别人（或换台电脑的你）复现？答案是 `requirements.txt`。

**导出**当前环境的所有依赖：

```bash
pip freeze > requirements.txt
```

生成的文件形如：

```
numpy==2.1.0
pandas==2.2.0
matplotlib==3.9.0
```

**安装**依赖清单里的所有库（换新环境时）：

```bash
pip install -r requirements.txt
```

这就是本教程根目录 `requirements.txt` 的作用——一条命令装齐第三、四部分需要的所有库。

> 标准工作流：新建项目 → 建虚拟环境并激活 → `pip install` 需要的库 → `pip freeze > requirements.txt` 记录下来。

## 18.5 规范的项目结构

一个小型 Python 项目通常长这样：

```
my_project/
├── .venv/                # 虚拟环境（不提交 git）
├── .gitignore            # git 忽略清单
├── README.md             # 项目说明
├── requirements.txt      # 依赖清单
├── src/                  # 源代码
│   ├── __init__.py
│   ├── main.py           # 程序入口
│   └── utils.py          # 工具函数
└── tests/                # 测试代码
    └── test_utils.py
```

关键点：源代码集中放（如 `src/`），入口清晰（`main.py` 里用 `if __name__ == "__main__":`），依赖和说明齐全。良好的结构让项目易读、易维护、易协作。

## 18.6 PEP 8：代码风格规范

**PEP 8** 是 Python 官方的代码风格指南。遵守它能让你的代码整洁、专业、易读。核心几条：

- **缩进**用 4 个空格，不要用 Tab 混排。
- 运算符两侧、逗号后加空格：`x = a + b`、`f(a, b)`。
- 命名：变量/函数用 `snake_case`，类用 `CamelCase`，常量用 `ALL_CAPS`。
- 每行不宜过长（一般 ≤ 79 或 99 字符）。
- 顶层函数/类之间空两行，方法之间空一行。
- import 放文件顶部，标准库、第三方库、本地模块分组。

可以用工具自动检查和格式化：

```bash
pip install black flake8      # black 自动格式化，flake8 检查风格
black your_file.py            # 一键格式化
flake8 your_file.py           # 检查不规范之处
```

> VS Code 里安装 Python 扩展后，可以配置“保存时自动用 black 格式化”，非常省心。

## 18.7 类型注解（Type Hints）入门

Python 是动态类型语言，但从 3.5 起支持**类型注解**——给参数和返回值标注类型。它不影响运行，但能提升可读性、让编辑器给出更好的提示和错误检查：

```python
def greet(name: str, times: int = 1) -> str:
    return (f"你好，{name}！" ) * times

# 变量也能注解
age: int = 18
scores: list[int] = [88, 92, 79]

def average(nums: list[float]) -> float:
    return sum(nums) / len(nums)
```

- `name: str` 表示 name 应该是字符串。
- `-> str` 表示函数返回字符串。

类型注解只是“提示”，写错类型 Python 也不会报错（除非用 `mypy` 这类工具检查）。但在稍大的项目里，它能显著减少 bug、提升协作效率。数据科学库的文档也大量使用它，学会看懂很有必要。

## 本章小结

- `pip install/uninstall/list/show` 管理第三方库；`python -m pip` 最稳妥。
- **虚拟环境**隔离项目依赖：`python -m venv .venv` 创建，`activate` 激活，`deactivate` 退出。
- `pip freeze > requirements.txt` 导出依赖，`pip install -r requirements.txt` 复现环境。
- 规范项目结构：源码、依赖、说明、测试各就各位。
- **PEP 8** 风格：4 空格缩进、合理空格与命名；用 `black`/`flake8` 自动化。
- **类型注解** `name: str -> int` 提升可读性与工具支持。
